## ✨ Возможности

### 🤖 Основные возможности ИИ
- 💬 **Интеллектуальный чат** – умная маршрутизация запросов (быстрые модели для простых, мощные для сложных)
- 🧠 **Продвинутое рассуждение** – выделенная модель для вычислений, генерации кода, творчества
- 🔍 **Мультимодальный анализ** – загрузка изображений и вопросы по их содержанию (llama.cpp + mmproj)
- 🎨 **Генерация изображений** – создание изображений из текста с автоматической оптимизацией промптов (модель Z-Image-Turbo)
- ✏️ **Редактирование изображений** – загрузите изображение и попросите изменить его (модель Flux.2 Klein 4B)
- 🎬 **Генерация видео** – создавайте короткие видео из текста или изображения+текста с помощью LTX-Video 2B (дистиллированная, 8 шагов)
- 🎤 **Распознавание речи** – преобразование голосовых сообщений в текст через Whisper ASR (faster_whisper)
- 🗣️ **Синтез речи** – озвучивание ответов через Piper TTS (мужской и женский голоса на русском и английском)

### 📁 Управление документами и знаниями
- 📚 **RAG с Qdrant** – загрузка документов (PDF, DOC, DOCX, TXT) и вопросы по содержимому
- 🗂️ **Сеансы чата** – множество независимых диалогов с авто-озаглавливанием
- 💾 **Экспорт чатов** – сохранение диалогов в HTML-файлы с встроенными медиа

### 🏠 Интеграция с домом (опционально)
- 📹 **Видеонаблюдение** – запрос снимков с IP-камер и анализ через мультимодальные модели
- 🔐 **Контроль доступа** – гранулярные права доступа к камерам для каждого пользователя

### 🔒 Конфиденциальность и безопасность
- 🏠 **100% локально** – вся обработка на вашем оборудовании
- 🔐 **Аутентификация по сессиям** – безопасный вход с хешированием паролей (Werkzeug)
- 🛡️ **Контроль доступа к файлам** – файлы доступны только авторизованным пользователям
- 🧹 **Изоляция данных** – данные каждого пользователя строго разделены
- 🔑 **CSRF-защита** – защита от подделки межсайтовых запросов для всех форм
- 🚦 **Rate Limiting** – защита от перебора паролей (5 попыток/минуту)
- 🔒 **Безопасность сессий** – HttpOnly и SameSite cookies, secure flag для HTTPS
- 📝 **Audit Logging** – логирование попыток входа и действий администратора
- 🔐 **HMAC-подпись очереди** – задачи Redis очереди подписаны для защиты от подделки
- 🛡️ **Валидация ввода** - строгая проверка пользовательского ввода (логины, пароли, параметры моделей) для предотвращения инъекций и некорректных данных.

### 👥 Пользовательский опыт
- 🌐 **Мультиязычность** – полный интерфейс и ответы ИИ на русском и английском языках
- 🌓 **Тёмная/светлая тема** – переключение тем с сохранением предпочтений

- 📎 **Вложения** – изображения, аудио и документы в диалогах

### ⚙️ Администрирование

## 🏗️ Архитектура

ПЛИИ — модульное Flask-приложение, оркестрирующее сервисы на экосистеме llama.cpp.

### Что нового в v8.6

| v8.6 (Новое) | Примечания |
|--------------|------------|
| 🎬 **Генерация видео** | Text-to-video и image+text-to-video через LTX-Video 2B distilled. Отдельный GPU-контейнер с изоляцией VRAM, T5 кодировщик на CPU (~8.9 GiB экономии VRAM). 8 шагов инференса, ~11с для 9 кадров 320×512. Требуется PyTorch с поддержкой sm_120 для RTX 5060 Ti (cu128 nightly). Включить через `--profile with-video`. |
| 🔄 **Retry при 502 для Qwen3VL** | Автоматический повтор (1 попытка, 5с задержка) при ответе 502 от мультимодальной модели. Circuit breaker предотвращает каскадные сбои. |
| 🧹 **Очистка CUDA после видео** | `torch.cuda.synchronize()`, `torch.cuda.empty_cache()`, `gc.collect()` в ltx_wrapper.py после каждой генерации. Повторная выгрузка LLM через `unload_llamacpp_model()` в `finally` блоке video.py. |
| 📐 **Единый ресайз (1536px)** | Все загружаемые изображения уменьшаются до **1536px** по длинной стороне (`MAX_IMAGE_SIZE`). Предотвращает переполнение контекста Qwen3VL (было 3840×2160 → ~5300 токенов, теперь ~950). Отдельный `_resize_for_classify()` больше не нужен. |
| 🐛 **llama-swap обновлён до v217** | Образ `ghcr.io/mostlygeek/llama-swap:cuda` обновлён с v212 (llama-server 9128) до v217 (llama-server 9294). Включает PR #18361 (Blackwell native builds fix) и PR #22522 (PDL для Hopper+). Исправляет краши Qwen3VL на RTX 5060 Ti (sm_120). |
| 🖼️ **Ресайз для редактирования (1024px)** | Исходные изображения перед SD inpainting уменьшаются до **1024px** по длинной стороне (было без ограничений, риск OOM). |
| 📊 **Диагностика GPU памяти** | Новый метод `log_gpu_memory()` в resource_manager логирует состояние VRAM через llama-swap API или nvidia-smi. Вызывается после генерации видео для проверки очистки. |
| ⚡ **Оптимизация загрузки чатов** | `file_data` удаляется из JSON в `content` в `get_session_messages()`, если файл на диске (`file_path` IS NOT NULL). Уменьшает размер ответа ~1000× для сессий с изображениями (10 картинок: ~15 MB → ~10 KB). Аудио без `file_path` не затрагивается. |
| 🖼️ **Сохранение пропорций видео** | Видео из изображения теперь использует aspect ratio исходной картинки. Широкие (w/h > 1.2) → 896×512, высокие (w/h < 0.8) → 512×896, квадратные → 512×512. Реализовано в `generate_video_params_from_image()`. |


### Основные компоненты

| Компонент | Назначение | Технология | Порт |
|-----------|------------|------------|------|
| **Flask Web** | Веб-интерфейс, маршрутизация, API | Python | 5000 |
| **llama-swap** | Динамическое управление и маршрутизация LLM-моделей (прокси для llama.cpp) | Go + llama.cpp | 8080 |
| **stable-diffusion.cpp** | Генерация изображений (Z_image_turbo) и редактирование (Flux.2 Klein 4B) | C++ + CUDA | 7861 |
| **LTX-Video** | Генерация видео (text-to-video / image+text-to-video) | Python + PyTorch | 7872 |
| **Whisper ASR** | Распознавание речи | faster_whisper | 9000 |
| **Piper TTS** | Синтез речи | ONNX + Piper | 8888 |
| **Qdrant** | Векторная база данных для RAG | Rust | 6333 |
| **Redis** | Управление очередью запросов | C | 6379 |
| **PostgreSQL** | Учётные записи, сессии, сообщения | SQL | 5432 |
| **Менеджер ресурсов** | Адаптивное управление GPU/CPU/RAM, предотвращает ошибки нехватки памяти, координирует доступ к GPU | Python |
| **Circuit Breaker** | Предотвращает каскадные сбои, временно блокируя вызовы к проблемным сервисам (llama.cpp, sd.cpp, Whisper) после повторяющихся ошибок | Python |

### Архитектура одиночного сервера

Все сервисы работают на одной машине с разделением GPU:

```text
┌───────────────────────────────────────────────────────────────┐
│              FLAI Web (Flask)                        │
│           Очередь Redis → Маршрутизатор → Ответ               │
└──────┬──────────┬────────────┬──────────────┬─────────────────┘
       │          │            │              │
       ▼          ▼            ▼              ▼
   llama-swap  sd.cpp      LTX-Video    Whisper/Piper/Qdrant
   :8080       :7861       :7872        (отдельные контейнеры)
   (динамическая маршрутизация моделей через llama-swap)
```

**Динамическая маршрутизация моделей**: llama-swap выступает прокси для llama.cpp, динамически загружая/выгружая GGUF-модели по запросу. В VRAM одновременно находится только одна модель, переключение происходит автоматически на основе типа запроса. Конфигурация моделей управляется через панель администратора и хранится в базе данных.

> 🎬 **Генерация видео** использует **отдельный GPU-контейнер** (`ltxvideo`) с собственной VRAM-областью. Перед каждой генерацией видео LLM-модель llama.cpp автоматически выгружается из VRAM для освобождения памяти под видео-пайплайн (трансформер + VAE ≈ 6 GiB). После генерации кэш CUDA очищается, а LLM-процессы повторно выгружаются для чистого состояния GPU. T5 text encoder остаётся на CPU для экономии VRAM.

## 📋 Системные требования

### Рекомендации по оборудованию
| Компонент | Минимум | Рекомендуемое | Оптимальное |
|-----------|---------|---------------|-------------|
| **ОЗУ** | 16 ГБ | 32 ГБ | 32+ ГБ |
| **ЦПУ** | 4 ядра | 4+ ядер | 8+ ядер |
| **GPU** | NVIDIA 12 ГБ VRAM | NVIDIA 16 ГБ VRAM | NVIDIA 24+ ГБ VRAM |
| **Диск** | 40 ГБ | 80+ ГБ SSD | 100+ ГБ SSD NVMe |

> **Примечания по GPU:**
> - **8 ГБ** достаточно для чата + эмбеддингов (без мультимодальной, без генерации изображений/видео).
> - **12 ГБ** — мультимодальная модель (Qwen3VL ~6 ГБ) или генерация изображений (Z_image_turbo ~7 ГБ в отдельном контексте), но не одновременно.
> - **16 ГБ** — мультимодальная + рассуждения + изображения + видео через переключение VRAM (только 1 LLM-модель загружена; SD и видео в отдельных GPU-контекстах с авто-выгрузкой LLM).
> - **24+ ГБ** — мультимодальная и рассуждения одновременно в VRAM без выгрузки.
>
> **Режим без GPU:** Чат, рассуждения, эмбеддинги, RAG и TTS работают без GPU (медленнее). Генерация изображений работает, но 10–30 минут на картинку. Мультимодальный анализ (зрение) и генерация видео требуют GPU.

### Программное обеспечение
- Linux-сервер с **NVIDIA GPU** (требуется поддержка CUDA)
- Установлены **драйверы NVIDIA** на хосте
- Установлен **NVIDIA Container Toolkit**
- Docker Engine ≥ 20.10
- Docker Compose ≥ 2.0
- Интернет (только для первоначальной загрузки моделей)

> 💡 **Примечание**: После загрузки GGUF-моделей ПЛИИ работает полностью офлайн.

---

### 💻 Запуск без GPU (режим CPU)

ПЛИИ может работать на серверах без GPU благодаря автоматическому определению в скрипте развёртывания. Если видеокарта NVIDIA не обнаружена, скрипт использует образы для CPU. Производительность будет заметно ниже, но все функции останутся доступны.

- Чат и рассуждения: работают, но медленнее в 3–10 раз.
- Генерация изображений: работает, но одно изображение может генерироваться 10–30 минут.
- Обработка голоса (Whisper, Piper) и поиск по документам (RAG) не затрагиваются.

Для принудительного запуска в режиме CPU даже при наличии GPU выполните:
```bash
docker compose -f docker-compose.cpu.yml --profile with-image-gen --profile with-voice --profile with-rag up -d
```

## 🚀 Быстрый старт

> 💡 **Примечание**: На хосте должны быть установлены **драйверы NVIDIA** и **NVIDIA Container Toolkit**.

### Вариант A: Автоматическое развёртывание (рекомендуется)

Единый скрипт развёртывания управляет всем: настройкой окружения, загрузкой моделей, сборкой и запуском.

```bash
git clone https://github.com/barval/flai.git
cd flai

# Только чат + llama.cpp
./deploy.sh --download-models

# + Генерация/редактирование изображений
./deploy.sh --download-models --with-image-gen

# + Голос (Whisper ASR + Piper TTS)
./deploy.sh --download-models --with-image-gen --with-voice

# Всё включая RAG (Qdrant)
./deploy.sh --download-models --with-image-gen --with-voice --with-rag

# + Генерация видео (LTX-Video)
./deploy.sh --download-models --with-image-gen --with-voice --with-rag --with-video

# Запуск тестов после развёртывания
./deploy.sh --download-models --with-image-gen --run-tests
```

Также доступна русская версия скрипта:
```bash
./deploy-ru.sh --download-models --with-image-gen --with-voice --with-rag --with-video
```

### Вариант B: Ручное развёртывание

### 1. Клонирование и настройка

```bash

# Создать директории и назначить владельца

# Скопировать шаблон окружения

# Сгенерировать секретный ключ
sed -i "s|^SECRET_KEY=.*|SECRET_KEY=$(python3 -c "import secrets; print(secrets.token_hex(32))")|" .env

sed -i "s|^QDRANT_API_KEY=.*|QDRANT_API_KEY=$(python3 -c "import secrets; print(secrets.token_hex(32))")|" .env

# Отредактировать .env (часовой пояс, URL сервисов и т.д.)
nano .env
```

### 3. Сборка и запуск сервисов

```bash
# Только чат и рассуждения (без генерации изображений)
docker compose -f docker-compose.gpu.yml up -d

# С генерацией изображений
docker compose -f docker-compose.gpu.yml --profile with-image-gen up -d

# С голосовыми функциями
docker compose -f docker-compose.gpu.yml --profile with-voice up -d

# Полный стек: чат + изображения + голос + RAG
docker compose -f docker-compose.gpu.yml --profile with-image-gen --profile with-voice --profile with-rag up -d
```

> ⏱️ **Первая сборка занимает время**: stable-diffusion.cpp компилируется из исходников (~5-10 минут). Последующие сборки используют кеш Docker.

### 4. Установка пароля администратора


## 🎨 Генерация и редактирование изображений

### Модель генерации

Проект использует **Z_image_turbo** как единственную модель генерации:

| Модель | Шаги | CFG Scale | Разрешение | Примечания |
|--------|------|-----------|------------|------------|
| **Z_image_turbo** | 10 | 1.0 | до 1536×1536 | Быстрая, flow-matching |

Все загружаемые изображения автоматически уменьшаются до **1536px** по длинной стороне (настраивается через `MAX_IMAGE_SIZE` в `.env`) для предотвращения переполнения контекста Qwen3VL и уменьшения места на диске.

### Редактирование изображений (Flux.2 Klein 4B)

Загрузите изображение и попросите его изменить (например, «сделай зрачки зелёными», «убери второе солнце»). Система использует:
1. **Мультимодальную модель** (Qwen3VL) для анализа изображения и создания промпта редактирования
2. **Модель Flux.2 Klein 4B** через stable-diffusion.cpp для выполнения редактирования
3. Оригинальное изображение сохраняется, за исключением запрошенных изменений

Исходные изображения для редактирования автоматически уменьшаются до **1024px** по длинной стороне для предотвращения нехватки памяти на GPU 16GB. При уменьшении показывается системное уведомление с исходными и новыми размерами.

Редактирование использует отдельные файлы моделей и работает независимо от генерации — конфликтов между ними нет.

### Генерация видео (LTX-Video 2B)

Проект использует **LTX-Video 2B 0.9.8 distilled** для генерации видео:

| Модель | Шаги | Частота кадров | Разрешение | Примечания |
|--------|------|---------------|------------|------------|
| **LTX-Video 2B distilled** | 8 | 8–30 fps | до 768×1344 | Дистиллированная, один GPU (~6 ГБ VRAM) |

Генерация видео запускается в **отдельном GPU-контейнере** (через `--profile with-video`). Перед генерацией LLM-модель llama.cpp автоматически выгружается из VRAM. После генерации кэш CUDA очищается, а LLM-процессы повторно выгружаются для гарантии чистого состояния GPU. T5 text encoder (~8,9 ГБ в bf16) остаётся на CPU.

**Ресайз исходного изображения:** Изображения для image-to-video уменьшаются до **896px** по длинной стороне перед отправкой в LTX-пайплайн (снижает нагрузку на VRAM и сеть). Системное уведомление показывает исходные и новые размеры.

**Сохранение пропорций:** При генерации видео из изображения разрешение видео автоматически подстраивается под aspect ratio исходной картинки: квадратные → 512×512, широкие (w/h > 1.2) → 896×512 ландшафт, высокие (w/h < 0.8) → 512×896 портрет.

**Требуемые модели:**
1. `ltxv-2b-0.9.8-distilled.safetensors` (~5,9 ГБ) — диффузионный трансформер + VAE
2. `PixArt-alpha/PixArt-XL-2-1024-MS` text encoder / tokenizer — T5-XXL кодировщик (~18 ГБ на диске в float32, ~8,9 ГБ в VRAM в bf16)

```bash
# Загрузка через скрипт развёртывания
./deploy.sh --download-models --with-video

# Или вручную:
bash services/ltx_video/download-t5-encoder.sh
```

**Конфигурация:**
```bash
LTX_VIDEO_WRAPPER_URL=http://flai-ltxvideo:7872
LTX_VIDEO_MODEL=ltxv-2b-0.9.8-distilled
LTX_VIDEO_TIMEOUT=600
```

### Сборка stable-diffusion.cpp

Сервис `sd_cpp` **собирается из исходников** при первом запуске:
1. Клонирует `https://github.com/leejet/stable-diffusion.cpp`
2. Инициализирует git-подмодули (`ggml`, `thirdparty/*`)
3. Компилирует с CUDA 13.0.1 (`cmake -DSD_CUDA=ON`)
4. Создаёт бинарные файлы `sd-server` и `sd-cli`

> ⏱️ **Первая сборка**: ~5-10 минут в зависимости от ЦПУ. Последующие сборки используют кеш Docker.

## 📚 RAG (поиск по документам)

### 1. Настройка RAG в Админ-панели

После запуска сервисов войдите как администратор и перейдите в **Админ-панель → вкладка Модели**. Прокрутите вниз до секции **Чанки**. Здесь можно точно настроить поведение RAG:

- **Размер чанка (символов):** На сколько частей разбиваются документы при индексации.
- **Перекрытие чанков (символов):** Количество перекрывающихся символов между соседними чанками.
- **Стратегия чанкинга:** `fixed` (фиксированный размер) или `recursive` (рекурсивное разбиение по заголовкам/абзацам).
- **Число чанков (top_k):** Максимальное количество чанков, извлекаемых из Qdrant на один запрос.
- **Порог (документы):** Минимальная оценка схожести для обычных запросов к документам.
- **Порог (рассуждения):** Минимальная оценка схожести, когда RAG активируется из запроса на рассуждение.

Нажмите **Сохранить**, чтобы применить изменения. При изменении параметров чанкинга (размер или стратегия) автоматически запускается фоновая переиндексация всех документов.

> **Примечание:** Переменные окружения вроде `RAG_CHUNK_SIZE` в `.env` используются только как начальные значения до первого сохранения в админ-панели. Основная конфигурация хранится в базе данных.

### 💾 Резервное копирование и восстановление

В FLAI встроена система резервного копирования, доступная в Админ-панели на вкладке **Backups**.

**Типы резервных копий:**
- **Только пользователи:** Резервная копия таблицы `users` (учётные записи, права, настройки).
- **Полная:** Все данные: пользователи, сеансы чатов, сообщения, документы, загруженные файлы и конфигурации моделей.

**Операции:**
- **Создать:** Выберите тип и нажмите «Создать резервную копию». Архив сохраняется в `data/db_backups/`.
- **Восстановить:** Нажмите «Восстановить» на файле резервной копии, чтобы заменить текущие данные. *Внимание: существующие данные будут перезаписаны.*
- **Скачать:** Загрузить архив на локальный компьютер.
- **Удалить:** Удалить старые файлы резервных копий.

Резервные копии хранятся в виде `.tar.gz` архивов, содержащих SQL-дампы и каталоги с файлами. Восстановление требует подтверждения и записывается в журнал аудита.

- **Генерация и редактирование изображений** — создание изображений из текста через Z_image_turbo, редактирование загруженных через Flux.2 Klein 4B

- **Интеграция с камерами** — запрос снимков с IP-камер, анализ мультимодальными моделями, гранулярные права пользователей
- **Резервное копирование и восстановление** — полные копии или только пользователи из админ-панели (pg_dump + tar.gz)
- **CLI-инструменты** — `admin-password` для сброса пароля, `cleanup-uploads` для удаления orphaned файлов
- **Health check и метрики** — эндпоинт `/health` со статусом сервисов, `/metrics` для Prometheus
- **Переключатель стилей ответов** — выпадающее меню в шапке чата: нейтральный, академический, деловой, дружеский, весёлый
- **Штраф за повтор** — параметр `repeat_penalty` (1.0–2.0) для каждой модели, предотвращает зацикливание ответов
- **PostgreSQL 18** — обновление с 16 до 18 без потери данных
- **Форматирование служебных префиксов** — сообщения голоса, камеры, генерации/редактирования изображений показывают префикс жирным; исключены из TTS и буфера обмена
- **Миграция формата сообщений** — все старые служебные сообщения конвертированы в структурированный JSON `{prefix, text}`
- **SSE-доставка в реальном времени** — результаты очереди и новые сообщения доставляются через SSE (Redis pub/sub), полностью заменяя HTTP-пуллинг
- **Cache-busting статики** — все JS/CSS ассеты загружаются с `?v=timestamp`, исключая кеширование устаревших файлов после обновлений
- **CLI-команда** — `flask migrate-messages-format` для конвертации старых служебных сообщений в JSON-формат (поддерживает `--dry-run`)
- **Извлечение PDF через pdftotext** — точное сохранение позиционирования текста из сложных PDF (резюме hh.ru, таблицы, многоколоночная вёрстка)
- **SSE-событие индексации документа** — список документов обновляется автоматически при завершении или сбое индексации, без перезагрузки страницы
- **Надёжность SSE** — исправлено 4 корневых причины доставки голосовых сообщений (видимость иконки молнии, восстановление после разрыва, передача `user_id` для событий `message_new`)
- **Миграция `--add-emojis`** — `flask migrate-messages-format --add-emojis` для добавления `🎨` к старым сообщениям изображений (поддерживается `--dry-run`)
- **Адаптивная вёрстка для планшетов** — медиа-запрос для 769–1199px исправляет перекрытие подвалом строки ввода из-за разницы `100vh` и `100%` в мобильных браузерах
- **Исправление стриминга изображений** — токены после маркера `[-IMAGE-EDIT-]` не отбрасываются при SSE-доставке; устранена ошибка пустого запроса редактирования
- **GPU/CPU автоопределение для SD** — `sd_wrapper.py` определяет CUDA в контейнере; флаги offload CPU опущены на GPU; без `--cuda`; авт. откат на CPU
- **Переводы ошибок SD восстановлены** — `_sd_error_translation_markers()` в `utils.py` для pybabel; 8 ключей `.po` реактивированы
- **Исправление переключения сессий** — `chat-sessions.js`: вызов `loadMessages()` после удаления сессии; клик по активной — перезагрузка сообщений
- **Глобальный перевод ошибок** — все пользовательские ошибки обёрнуты в `_()`/`gettext()`; 14 новых ключей `.po`; правило в `AGENTS.md`
- **Исправление гонки ⚡ для аудио** — исправлена гонка `clearSessionQueue`/`fetchQueueStatus` для HTTP-аудио без `request_id`; единая иконка ⚡
- **Восстановление после F5** — ⚡, стриминг и финальный ответ выживают после обновления страницы; `onStreamToken`/`onResultCompleted` обрабатывают отсутствие `pendingRequestIds`
- **Монитор VRAM и автодеградация моделей** — фоновый опрос через `nvidia-smi` каждые 60 сек; прогрессивная деградация (100%→0% n_gpu_layers в 4 шага) при OOM; безопасные лимиты для каждого объёма VRAM
- **VRAM-калькулятор в админ-панели** — эндпоинт `/admin/api/model-estimate` оценивает VRAM (веса + KV cache + compute); авто-расчёт `n_gpu_layers` со слайдером
- **Offload-система для SD** — 4 уровня выгрузки VRAM (0=полный GPU → 3=полный CPU); проверка запаса VRAM (500MB) перед генерацией
- **Скорость токенов в реальном времени** — оценка токенов/сек во время стриминга; финальная скорость в заголовке сообщения; `completion_tokens` сохраняется в БД
- **Исправление кеша конфигурации моделей** — TTL-кеш заменён на версионирование по `updated_at`; устранена рассинхронизация между воркерами gunicorn
- **Исправление отображения архитектуры** — декодирование numpy byte-string (`[113 119 101 110 51]` → `qwen3`) в админ-панели
- **Расширение метаданных GGUF** — `parameter_count`, `head_count`, `head_count_kv`, `key_length`, `value_length` сканируются и сохраняются в БД
- **Генерация видео (LTX-Video 2B)** — text-to-video и image+text-to-video. Отдельный GPU-контейнер с изоляцией VRAM, T5 encoder на CPU, авто-выгрузка LLM llama.cpp. 8 шагов дистиллированного инференса, ~11с для 9 кадров 320×512.
- **Retry при 502 для Qwen3VL** — автоматический повтор (1 попытка, 5с задержка) при ответе 502 от мультимодальной модели. Circuit breaker предотвращает каскадные сбои.
- **Очистка CUDA после видео** — `torch.cuda.empty_cache()` + `gc.collect()` в ltx_wrapper.py в `finally` блоке. Повторная выгрузка LLM в video.py для удаления зомби-процессов.
- **Диагностика GPU памяти** — метод `log_gpu_memory()` через llama-swap API или nvidia-smi. Логируется после каждой генерации видео.
- **Единый ресайз изображений (1536px)** — `resize_image_if_needed` изменён с ограничивающего прямоугольника (3840×2160) на длинную сторону (1536px). Предотвращает переполнение контекста Qwen3VL.
- **Ресайз для редактирования (1024px)** — исходные изображения для SD inpainting уменьшаются до 1024px по длинной стороне для предотвращения OOM.
- **llama-swap обновлён до v217** — образ с llama-server 9294, включающий исправления крашей на Blackwell (sm_120).
- **Оптимизация загрузки чатов** — base64 `file_data` удаляется из JSON в `content` в `get_session_messages()`, если файл сохранён на диске. Уменьшает ответ API ~1000× (10 картинок: ~15 MB → ~10 KB).
- **Сохранение пропорций видео из изображения** — разрешение видео соответствует aspect ratio исходного изображения (широкие → 896×512, высокие → 512×896, квадратные → 512×512). Реализовано в `generate_video_params_from_image()`.

## 🧪 Тестирование

ПЛИИ включает комплексное тестирование всех ключевых компонентов и нагрузочное тестирование веб-интерфейса.

### Модульные тесты

```bash
# Установить зависимости для тестов
pip install -e ".[test]"

# Запустить все тесты
pytest

# С отчётом покрытия
pytest --cov=app --cov=modules --cov-report=html

# По маркерам
pytest -m unit                           # только unit-тесты (без внешних зависимостей)
pytest -m "not slow"                     # пропустить медленные тесты
pytest -m "not (requires_db or requires_redis)"  # без БД и Redis

# Отдельные файлы
pytest tests/test_backups.py
pytest tests/test_admin_routes.py
pytest tests/test_sd_cpp_module.py
pytest tests/test_queue.py
pytest tests/test_security.py
pytest tests/test_resource_manager.py
pytest tests/test_llama_swap_config.py
pytest tests/test_validators.py
pytest tests/test_model_config.py
```

> **Примечание**: `tests/conftest.py` использует in-memory mock базу данных по умолчанию (PostgreSQL не требуется). В CI доступна реальная PostgreSQL через переменную `DATABASE_URL`.

### Нагрузочное тестирование

Нагрузочные тесты используют [Locust](https://locust.io/) для эмуляции одновременных пользователей.

```bash
# Установить Locust (если ещё не установлен)
pip install locust

# Веб-интерфейс — открыть http://localhost:8089
locust -f tests/load/locustfile.py --host http://localhost:5000

# Автоматический режим — 10 пользователей, запуск 2/с, 1 минута
locust -f tests/load/locustfile.py --headless -u 10 -r 2 --run-time 1m

# Через вспомогательный скрипт
./tests/load/run_load_test.sh --host http://localhost:5000 --users 10 --spawn-rate 2 --run-time 1m
```

Подробная инструкция — в [tests/load/README-ru.md](tests/load/README-ru.md).
